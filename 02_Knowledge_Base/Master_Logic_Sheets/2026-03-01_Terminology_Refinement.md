# Master Logic Sheet: JSTQB 厳格定義の克服 (2026-03-01)

本日ミスがあった「紛らわしい用語」の境界線を整理します。ここを固めるだけでスコアは 90% を超えます。

## 1. 静的テスト：レビュー種別のマッピング
| 種別 | 主導者 | 主要目的 | 形式性 |
| :--- | :--- | :--- | :--- |
| **インフォーマル** | (任意) | 安価に欠陥を発見する | 最低 |
| **ウォークスルー** | **作成者** | **理解を合わせる**、学習、欠陥発見 | 低〜中 |
| **テクニカル** | 修練した司会 | 総意（合意）の形成、技術的判断 | 中〜高 |
| **インスペクション** | **モデレーター** | **欠陥発見、プロセス改善**、メトリクス収集 | **最高** |

## 2. 第2章：テストの「性質」
*   **機能テスト (Functional)**: 
    *   **「何を (What)」** するか。機能仕様書がベース。
*   **非機能テスト (Non-Functional)**: 
    *   **「どのように (How)」** 動作するか。信頼性、効率性、保守性など。
    *   *注：本日の問21で「どのように」と回答されましたが、それは非機能テスト（品質特性）の領域です。*

## 3. 第1章：テストウェアの呼称
*   **テストオラクル (Test Oracle)**: 
    *   **期待結果** を特定するための「ソース（正解源）」。
*   **テストベース (Test Base)**: 
    *   テストケースの **作成元** となる全てのドキュメント。

## 4. 第2章：修正後のテスト
*   **確認テスト (Confirmation Testing)**: 
    *   **「直したはずのバグ」** が本当に直ったかを確認。
*   **リグレッションテスト (Regression Testing)**: 
    *   **「関係ない場所」** に悪影響が出ていないかを確認。

## 5. 第5章：独立性の順序 (ISTQB/JSTQB定義)
1. **最高**: 外部組織（アウトソーシング）
2. **高**: 自組織内の独立したテスト専任チーム（QA部など）
3. **中**: 開発以外のメンバ（隣のチームの開発者など）
4. **低**: 開発者自身
