# TA Techniques Lab (AI-Driven QA Cheat Sheet)

ここでは、JSTQB AL TAシラバスの高度なテスト設計ロジックや実務上の制約を、LLM（AI）と連携して解決するためのプロンプトテンプレートやプラクティスを蓄積します。

---

## 1. テストプロセス制御（第1章）
### 【AI連携パターン】仕様不完全プロジェクトにおけるハイブリッドテスト設計
仕様変更が激しくドキュメントが不完全な場合、AIにテスト条件と高レベルテストケース（論理設計）の作成を支援させ、人間がドメイン知識を用いて低レベルテストケース（具象化・手順）および探索的テストを設計するアプローチをとる。

#### AIプロンプトテンプレート：高レベルテストケース（論理設計）のドラフト生成
```markdown
あなたは優秀なテストアナリスト（JSTQB AL TA）です。
提供する不完全な機能仕様（ドラフト）から、「テスト条件」と「高レベルテストケース（具象値を含まない論理テストケース）」を設計してください。

## 入力仕様
[ここに仕様書の内容をペースト]

## 出力フォーマット
### 1. 特定されたテスト条件（何をテストすべきか）
- TC-01: [テスト条件名]
- TC-02: [テスト条件名]

### 2. 高レベルテストケース（どうやってテストすべきか：論理設計）
- H-TC-01:
  - 対象テスト条件: TC-01
  - 事前条件: [論理的な事前状態]
  - 入力条件: [論理的な入力パラメータの範囲や組み合わせ条件。※具体的な値は書かないこと]
  - 期待結果: [論理的に期待されるシステム挙動。※具体的な画面文言やデータ値は書かないこと]
```

## 2. ライフサイクルにおける協調（第2章）
### 【AI連携パターン】TAとTTAの協調による負荷テスト期待結果の自動生成
TTAが構築する負荷テストシナリオに対して、TAが持つビジネスドメインやユーザー体験の視点（閾値）をLLMで抽出し、テスト実行時の「期待結果（合否判定）」にマッピングする。

#### AIプロンプトテンプレート：ユーザーレビューからのパフォーマンス許容閾値抽出
```markdown
あなたはゲーム/アプリの品質管理を担当するシニアQAマネージャーです。
過去のユーザーレビューや不満フィードバックのテキストから、ユーザーがストレスを感じる「パフォーマンス（応答速度、遅延、切断頻度）」に関する定性的な不満を分析し、TTA（テクニカルテストアナリスト）が負荷テストの判定基準として使用できる「定量的閾値（期待結果）」のドラフトを提案してください。

## ユーザーの声（ログ）
[レビューや問い合わせログをペースト]

## 出力フォーマット
### 1. ユーザーペインポイントの分析
### 2. 負荷テストの期待結果（判定閾値）の提案
- マッチング待機時間の最大閾値: [例: 5.0秒以内（それを超えると離脱リスク急増）]
- サーバー応答時間 (95パーセンタイル): [例: 500ms以内]
- エラーレート許容値: [例: 0.1%未満]
```

## 3. テスト技法（第3章）
### 【AI連携パターン】決定表（デシジョンテーブル）の厳密な最小化
LLMに仕様書から決定表を生成・圧縮させる際、論理的な誤りを防ぐためにマージ条件を明示する。

#### AIプロンプトテンプレート：決定表の設計と正しい最小化
```markdown
あなたは優秀なテストアナリスト（JSTQB AL TA）です。
以下の仕様から決定表（デシジョンテーブル）を生成してください。その際、列（ルール）の最小化（圧縮）は以下の厳密なルールに従ってください。

## 最小化（マージ）のゴールデンルール
1. 2つの列をマージしてある条件をドンケア「-」にできるのは、「他のすべての条件およびその結果としてのアクションが完全に一致しており、特定の1つの条件値だけが Y と N で異なっている」場合に限られます。
2. アクションが同一であっても、2つ以上の条件値が異なっている列同士は、安易にマージしないでください（他のルールを破壊するため）。

## 入力仕様
[ここに仕様書の内容をペースト]

## 出力フォーマット
### 1. マージ前の完全な決定表
### 2. マージされた決定表（ドンケア「-」を含む）
### 3. マージプロセスの説明（どのルールとどのルールをどの条件の違いでマージしたか）
```

### 【AI連携パターン】ペアワイズ法のテストケース数および組み合わせの検証
LLMが提案するペアワイズのテストケース数が妥当であるかを検証するためのガードレール。

#### AIプロンプトテンプレート：ペアワイズ検証
```markdown
提供するパラメータ（因子）と値（水準）のリストについて、ペアワイズ法（オールペア法）を適用したテストケースを生成してください。
また、生成したテストケース群が以下の検証ルールを満たしているか、ステップバイステップで自己検証してください。

## 検証ルール
- **論理的下限の確認**: 最大の水準数を持つ上位2因子の水準数の積（例: 因子Aが4水準、因子Bが4水準なら 4×4=16）を計算し、生成されたテストケース数がこの下限値以上になっていること。
- **全ペアの網羅確認**: 任意の2因子について、すべての値の組み合わせ（ペア）が少なくとも1回はテストケース中に現れていること。

## 入力パラメータ
[因子と水準のリストをペースト]
```
