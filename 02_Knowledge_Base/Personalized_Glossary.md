# Personalized Glossary: JSTQB 攻略・自分専用用語集

このドキュメントは、演習で迷った用語や、JSTQB特有の「ひっかけ」ポイントを蓄積するためのものです。
毎回のセッション終了後にアップデートし、試験直前の最強の復習ツールとして活用します。

---

## 📖 第1章：テストの基礎
| 用語 | JSTQB的定義・ひっかけ回避ポイント | 覚え方・対比 |
| :--- | :--- | :--- |
| **テストオラクル** | **期待結果**を確認するための情報源（仕様書、既存システム、知識など）。正解の「基準」そのもの。 | × テストベース（作成元） |
| **テストベース** | テストケースやテストスイートの**作成元となる全てのドキュメント**。要件定義書、設計書など。 | 作成の「材料」 |
| **欠陥なしの証明不可** | テストをいくらやっても「欠陥がないこと」は証明できない。欠陥があることは示せるが、ないことは示せない。 | 原則1 |
| **殺虫剤のパラドックス** | 同じテストを繰り返すと新しい欠陥が見つからなくなる。定期的な見直しが必要。 | 原則5 |

## 📖 第2章：SDLC全体を通してのテスト
| 用語 | JSTQB的定義・ひっかけ回避ポイント | 覚え方・対比 |
| :--- | :--- | :--- |
| **確認テスト** | 別名：**再テスト (Retest)**。「修正が正しく行われたか」をピンポイントで確認する。 | 点のテスト |
| **リグレッション** | 変更によって**「他の場所」**に悪影響（デグレード）が出ていないかを確認する。 | 面のテスト |
| **機能テスト** | ソフトウェアが**「何を (What)」**するかを検証する。機能仕様に基づく。 | ⇔ 非機能テスト |
| **非機能テスト** | ソフトウェアが**「どのように (How)」**動作するか（性能、信頼性、保守性など）を検証する。 | 品質特性 |
| **保守テスト** | **運用開始後**の変更、移行、退役がトリガー。開発中の修正は含まない。 | 運用フェーズ |

## 📖 第3章：静的テスト
| 用語 | JSTQB的な役割・核心 | 形式性 |
| :--- | :--- | :--- |
| **インフォーマル** | 定義された手順なし。安価に欠陥を発見。 | 低 |
| **ウォークスルー** | **作成者**がリード。参加者の**理解を合わせる**ことが主目的。 | 低〜中 |
| **テクニカル** | 修練した司会。**技術的な合意（総意）**を形成する。 | 中〜高 |
| **インスペクション** | **モデレーター**（訓練された司会）がリード。**プロセス改善**とメトリクス収集も行う。 | **最高** |

## 📖 第4章：テスト分析と設計
| 用語 | JSTQB的定義・ひっかけ回避ポイント | 覚え方・対比 |
| :--- | :--- | :--- |
| **テスト分析** | **「何を (What)」**テストするか。テスト条件を特定する活動。 | 分析 ＝ 条件 |
| **テスト設計** | **「どのように (How)」**テストするか。**高水準ケース**作成、期待結果特定。 | 設計 ＝ ケース作成 |
| **テスト実装** | **実行準備**。低水準ケース、テストスイート、実行順序、データのロード。 | 実装 ＝ 準備完了 |
| **判定網羅** | 全ての判定出口（T/F）を1回は通る。1つのケースで複数の判定を通れる。 | 判定の「出口」 |
